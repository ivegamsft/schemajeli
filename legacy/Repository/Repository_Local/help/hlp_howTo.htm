What is a Repository?

    The term "Repository" is very general and because it is used in many different areas, it makes it hard to distinguish between all of the definitions.  When the EDW team refers to a Repository, we are referring to a new way of thinking about systems. We refer to how the systems themselves interact, and how we as users interact with them. 
To give the repository a solid and clear definition, consider some of the areas to which a repository tool can add value to enterprise meta-data management:

Business Processes

Business Processes include the development of enterprise models that document, analyze, or simulate the operational environment and its control and product flow. Repositories can act as a place to maintain the models, the the relationships between the models

Application Development

Application development is one of the main areas in which repositories have been used.    Software development is a multi-step process, typically involving several developers, users, departments, etc., sometimes in different geographic locations. Having common ideas about how a system is run and how it affects the business is difficult. Repositories help to bridge the gap between the business and the systems they use.

Data Modeling and Design

The task of data modeling and design typically involves multiple developers. A web-based repository allows querying of shared meta-data and takes on the role of data dictionary for the enterprise.

Data Warehousing

The area of Data Warehousing is still a mystery to most users.  The Repository stores the combination of technical and business meta-data which helps to assist users with definitions and explanations and makes the tasks of designing, managing, and using a data warehouse easier.

Legacy Application Maintenance

The maintainer of a legacy system can use a repository tool to document an existing system or to analyze the impact of modifications to that system. Today's enterprises base their day-to-day operation on computers from desktop to mainframe and have many deployed database systems. In many cases, the original developers of databases are unknown and specifications are unavailable. Nevertheless, such legacy systems sometimes need immediate, mission-critical modification. Retaining the database structure with all its dependencies in a repository allows the individual developer to explore the legacy system and to determine the impact of changes.

Why we need a Repository?

In the past, businesses have been challenged to find and consolidate all of the information needed for efficient decision-making. Capturing business knowledge such as goals, objectives, processes and rules, as well as terminology and categorizations, and enabling the automated linkage between a wide variety of business information types, is needed for the creation of enterprise information portals or "digital dashboards," which enable real-time decision-making by providing businesses with a single window into all of their information.

  • One representation
    a common model that allows systems to share information described in a repository and to promote a single description for the entire enterprise

  • One place
    data-management and integration technology that supports querying repository information and browsing the repository by being able to follow relationships between objects in the Repository.

Navistar meta-data Naming Standards

meta-data naming standards enforcement support the effective use of data warehousing, data mining, system engineering, data architecture and data analysis. The function of the Navistar Repository is:

  • To make information readily available, understandable, common, and functional
  • To establish and enforce naming rules and standards to promote a common language among systems
  • To enable sharing and increase reusability among teams and systems

Overview of the Navistar Repository

The Navistar Repository was designed with ease-of-use and functionality in mind. After you have logged in, the main screen will display any messages that are pertinent to the current session. Additional resource links are also displayed here in case you want to research other areas related to the repository. The navigation menu is displayed in the upper portion of your screen. You can select alternate menus by following the associated links on the menu. Once you select a menu option, the menu will change and the options in the drop down list will also change display the appropriate selections. Click the drop down box in the menu and when a selection is made, the main portion if your browser will display the screen. You can also press Go. Some screens allow you to navigate from within the main portion of the window.
The menu options are displayed according to your user authority. All users by default receive the following menu options. For Help on a menu option follow the link:

  • Search
    • Servers
    • Databases
    • Tables
    • Elements
    • Element Key Words
    • Standard Abbreviations
  • Reports
    • Relational Object
    • View Class Words
    • Servers
    • Generate DDL

Users with the ability to add and maintain objects receive in addition to the above:

  • Maintain
    • Database
    • Tables
    • Element

Administrators receive all of the above menu options including:

  • Administration
    • Server
    • Standard Abbreviation
    • Users
